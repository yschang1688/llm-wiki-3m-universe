# 提示工程

> 提示工程 (Prompt Engineering) 是設計和優化輸入提示以引導大語言模型 (LLM) 產生期望輸出的技術，是使用生成式AI的核心能力，也是 2026 年 AI 開發工作流程中不可或缺的基礎技能。

| 提出年 | 2023–2024 (系統化發展) |

## 重點

- **【核心發現】**提示工程的層次已從單一指令進化為「結構化設計模式」；透過思維鏈 (CoT) 與上下文學習 (ICL)，能將 LLM 從單純的對話機器人轉化為可預測的執行引擎。
- **本質**：透過精心設計的自然語言指令，在不修改模型參數的前提下，最大化模型的推理與生成能力
- **思維鏈 (Chain-of-Thought, CoT)**：在提示中要求模型「逐步推理」，顯著提升數學、邏輯、多步驟任務的準確率
- **上下文學習 (In-context Learning)**：在提示中嵌入少量輸入輸出示例 (Few-shot Examples)，讓模型推斷任務格式與規則
- **指令調優協同 (Instruction Tuning)**：配合經過指令微調的模型使用，清晰的指令格式能釋放更強效能
- **2026 年趨勢**：結構化提示設計（角色設定、思考框架、輸出格式規範）成為主流，提示本身可版本控制、測試、優化

## 細節

### 核心提示技術

#### 零樣本提示 (Zero-shot Prompting)

直接告知模型任務，不提供示例。適合簡單任務或已充分指令微調的模型：
```
將以下文字翻譯成英文，只回傳翻譯結果：
「今天天氣很好」
```

#### 少樣本提示 (Few-shot Prompting)

在提示中嵌入 2–8 個輸入輸出範例，引導模型學習格式與推理模式：
```
正面情緒 → 1
「今天很開心」 → 1
「這部電影很無聊」 → 0
「會議進行得很順利」 → ?
```

#### 思維鏈提示 (Chain-of-Thought, CoT)

加入「讓我們一步一步思考 (Let's think step by step)」或直接提供帶推理過程的示例，大幅提升複雜推理任務效能：

| 任務類型 | CoT 效益 | 2024 課程數據 (GPT-XR2) |
|---------|---------|-------------------------|
| 數學應用題 | 準確率提升 30–60% | 17.7% $\to$ **78%** |
| 多步邏輯推理 | 顯著降低跳步錯誤 | — |
| 程式除錯 | 引導模型逐行分析 | — |
| 常識問答 | 效益有限 | — |

**自動思維鏈 (Auto-CoT)**：由模型自動生成推理步驟，無需人工撰寫示例。叫模型「**先解釋理由，再給評分**」能顯著提升其在批過、審核等主觀判斷任務中的準確度，使機器輸出更接近人類專家共識。

#### 結構化提示設計 (Structured Prompt Design)

2026 年的主流格式，將提示分為明確的模塊：
```
## 角色
你是一位資深 Python 工程師。

## 任務
審查以下程式碼，指出潛在的效能問題。

## 輸出格式
以 JSON 格式回傳，欄位：[issue, severity, suggestion]

## 程式碼
{code}
```

### 提示工程的層次：輸入與輸出
在實務中，提示工程涵蓋兩個關鍵層次：
1. **輸入層面 (Input Level)**：撰寫適當的提示語，類似於機器學習中的特徵工程，決定了模型的基礎理解。
2. **輸出層面 (Output Level)**：調整生成參數（如 Temperature, Top-P, Top-K, Token 長度限制），以微調模型回應的多樣性與確定性。

### 提升 AI 系統能源效率的策略
隨著系統規模擴大，提示設計也需考慮性能與能耗：
- **提示快取 (Prompt Caching)**：利用 API 層面的快取機制，減少重複提示的運算能耗與成本。
- **批次處理 (Batch APIs)**：將大量請求打包處理，優化數據中心資源調度（如在再生能源供應充足時段運行）。
- **輕量化模型與邊緣運算**：針對簡單任務選用較小模型（如 Gemini Flash 級別），降低整體系統負載。

### 降低模型幻覺與偏誤的方法
1. **基於輸入的檢索 (RAG)**：結合[[檢索增強生成]]技術，使模型依據可驗證的外部事實進行回答，從根本上減少虛假資訊。
2. **內部自我修正 (Self-Correction)**：在提示中明確要求事實驗證步驟，引導模型糾正自身的邏輯或事實錯誤。
3. **評估偏誤緩解**：選用先進模型作為自動評分器時，透過**多重抽樣**與**順序翻轉**（翻轉選項順序再次詢問）來緩解模型本身可能存在的固定偏誤。

### 進階技術

**自我一致性 (Self-Consistency)**：同一提示生成多個推理路徑，取多數答案，提升可靠性。

**提示鏈 (Prompt Chaining)**：將複雜任務分解為多個子提示，前一步驟的輸出作為下一步的輸入，適合多階段工作流程。

**角色扮演提示 (Role Prompting)**：指定模型扮演特定專家角色，引發更專業的回應風格與知識深度。

### 實務應用場景

#### 搭配 AI Agent 的 Vibe Coding

在 [[n8n工作流程]] 中，提示工程用於：
- 定義 AI Agent 的角色、目標與行為邊界
- 設計工具呼叫 (Tool Calling) 的判斷提示
- 控制多 Agent 協作中的溝通格式

Vibe Coding 場景（Cursor、GitHub Copilot）中，有效提示包含：①明確的函式簽章、②預期行為描述、③邊界條件說明，可大幅提升程式生成品質。

#### 與 RAG 結合

[[檢索增強生成]] (RAG) 中，提示工程負責：
- 設計查詢改寫提示（將用戶問題轉為更適合向量搜索的形式）
- 設計上下文整合提示（告訴模型如何使用檢索到的文件片段）
- 防止「忽略上下文」問題（明確指示模型僅依據提供文件回答）

#### IPAS 考試核心技能

[[IPAS AI應用規劃師]] 初級考試中，提示工程是生成式AI應用的核心考點，包含：
- 辨識有效 vs. 無效提示的差異
- 了解 CoT、Few-shot 的適用場景
- 提示安全：防止提示注入 (Prompt Injection) 攻擊

### 提示工程的局限性

| 局限 | 說明 | 替代方案 |
|-----|------|---------|
| 效果依賴模型能力 | 弱模型再好的提示也有上限 | 選用更強模型或微調 |
| 不穩定性 | 微小措辭差異導致輸出大相徑庭 | 建立提示測試套件 |
| 上下文視窗限制 | 太長的提示超出模型處理範圍 | RAG 或長上下文模型 |
| 無法持久化知識 | 每次對話重新提供背景 | 微調 (Fine-tuning) |
